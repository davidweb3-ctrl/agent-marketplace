# UX Flows for AI Agent Web3 Marketplace

Below are the detailed UX flows in structured Markdown format. Each flow includes user steps, UI states, triggered API calls, possible errors with handling, and loading states. Flows are based on a Web3 marketplace where providers onboard AI agents, clients create missions, and disputes/offboarding are managed. Assumptions: AGNT is the native token; escrow uses smart contracts; IPFS for data storage; Stripe for fiat; wallets for crypto.

## Flow 1: PROVIDER ONBOARDING
Provider connects wallet, stakes 1000 AGNT, submits agent card, uploads to IPFS, performs on-chain tx, and gets listing confirmation.

### Step 1: Connexion Wallet
- **User Sees (UI Component)**: Button "Connect Wallet" on onboarding page. Modal pops up with wallet providers (e.g., MetaMask). After connection, displays connected address and balance.
- **Backend (API + Contract)**: API call to wallet provider SDK (e.g., Web3.js) to request connection. No contract interaction yet.
- **Error States & Messages**: If wallet not installed: "Please install MetaMask or compatible wallet." If connection rejected: "Wallet connection rejected. Please try again." Redirect to retry.
- **Loading States**: Spinner on button: "Connecting...".

### Step 2: Staking 1000 AGNT
- **User Sees (UI Component)**: Form showing current AGNT balance, input for amount (pre-filled 1000), "Stake" button. Progress bar if multi-step.
- **Backend (API + Contract)**: API checks balance via blockchain query. Triggers smart contract call to stake(1000 AGNT) on AGNT token contract. Waits for tx confirmation.
- **Error States & Messages**: Insufficient balance: "You need at least 1000 AGNT to stake. Current balance: X." Tx failure (e.g., gas): "Transaction failed. Please check gas and try again." Retry button.
- **Loading States**: Modal with spinner: "Staking in progress... Awaiting confirmation (1/3 blocks)."

### Step 3: Soumission Agent Card
- **User Sees (UI Component)**: Form with fields: agent name, description, skills, image upload. Preview card. "Submit" button.
- **Backend (API + Contract)**: API validates form data (e.g., required fields). No contract yet; stores temporarily in backend DB.
- **Error States & Messages**: Invalid input: "Description must be at least 50 characters." File too large: "Image must be under 2MB." Show inline errors.
- **Loading States**: Button spinner: "Validating submission...".

### Step 4: IPFS Upload
- **User Sees (UI Component)**: Upload progress bar after submission. "Uploading to IPFS" message.
- **Backend (API + Contract)**: API uploads agent card data (JSON + image) to IPFS via Pinata or similar service. Returns IPFS hash.
- **Error States & Messages**: Upload failure: "IPFS upload failed. Please try again or check connection." Retry option.
- **Loading States**: Progress bar: "Uploading... (XX%)".

### Step 5: Tx On-Chain
- **User Sees (UI Component)**: Confirmation modal: "Confirm On-Chain Listing" with tx details. Wallet prompt for signature.
- **Backend (API + Contract)**: Triggers smart contract call to marketplace contract: listAgent(IPFS_hash, provider_address). Includes staking reference.
- **Error States & Messages**: Tx rejected: "Transaction rejected by wallet." Gas limit: "Insufficient gas. Increase and retry." Show tx hash for manual check.
- **Loading States**: Spinner: "Processing transaction... Awaiting blockchain confirmation."

### Step 6: Confirmation Listing
- **User Sees (UI Component)**: Success page: "Agent Listed!" with agent card preview, link to marketplace. Dashboard redirect.
- **Backend (API + Contract)**: API listens for contract event (AgentListed) via websocket. Updates backend DB with listing status.
- **Error States & Messages**: Event timeout: "Confirmation delayed. Check blockchain explorer with tx hash: [hash]." Manual refresh button.
- **Loading States**: Spinner: "Waiting for listing confirmation...".

## Flow 2: CLIENT FIRST MISSION
Client discovers agent, fills mission form, commit-reveal, pays via fiat (Stripe) or crypto, escrow confirmation, mission start.

### Step 1: Découverte Agent
- **User Sees (UI Component)**: Marketplace grid with agent cards (search, filters). Click card to view details.
- **Backend (API + Contract)**: API fetches listed agents from backend DB (synced with on-chain data via contract queries).
- **Error States & Messages**: No agents: "No agents found. Try different filters." Network error: "Failed to load agents. Refresh page."
- **Loading States**: Skeleton cards: "Loading agents...".

### Step 2: Mission Form
- **User Sees (UI Component)**: Form on agent page: mission title, description, deadline, budget. "Create Mission" button.
- **Backend (API + Contract)**: API validates form (e.g., budget >0). Generates commit-reveal hash if needed.
- **Error States & Messages**: Invalid budget: "Budget must be at least $10." Missing fields: "Please fill all required fields."
- **Loading States**: Button spinner: "Preparing mission...".

### Step 3: Commit-Reveal
- **User Sees (UI Component)**: Modal: "Commit Mission" with blinded details. "Confirm" button.
- **Backend (API + Contract)**: API generates commit hash. Triggers contract call: commitMission(hash, agent_id).
- **Error States & Messages**: Commit failure: "Commit transaction failed. Try again." Reveal later in flow if needed.
- **Loading States**: Spinner: "Committing to blockchain...".

### Step 4: Fiat Payment (Stripe) OU Crypto (Wallet)
- **User Sees (UI Component)**: Payment selector: "Pay with Fiat (Stripe)" or "Crypto (Wallet)". For Stripe: card form. For crypto: wallet prompt with amount in USDC/ETH.
- **Backend (API + Contract)**: Stripe: API creates session, handles webhook for success. Crypto: Contract call to escrowMission(amount). Reveal after payment.
- **Error States & Messages**: Stripe decline: "Payment declined. Try another card." Crypto insufficient: "Insufficient funds in wallet." 
- **Loading States**: For Stripe: "Redirecting to Stripe...". Crypto: "Awaiting wallet confirmation...".

### Step 5: Escrow Confirmation
- **User Sees (UI Component)**: Success modal: "Payment Escrowed!" with mission ID.
- **Backend (API + Contract)**: Listens for EscrowCreated event. Updates DB status to "Escrowed".
- **Error States & Messages**: Event not received: "Escrow confirmation delayed. Check tx: [hash]."
- **Loading States**: Spinner: "Confirming escrow...".

### Step 6: Mission Start
- **User Sees (UI Component)**: Dashboard update: Mission status "Pending Provider Acceptance".
- **Backend (API + Contract)**: API notifies provider (e.g., via push/email). No further contract call.
- **Error States & Messages**: Notification fail: "Provider notification sent, but check status manually."
- **Loading States**: N/A (asynchronous).

## Flow 3: MISSION LIFECYCLE (Provider Side)
Provider gets notification, accepts, sets IN_PROGRESS, delivers, waits for approve/dispute.

### Step 1: Notification Nouvelle Mission
- **User Sees (UI Component)**: Dashboard notification bell. Pop-up or list: "New Mission: [title]" with details.
- **Backend (API + Contract)**: API pushes notification via websocket/event from contract (MissionEscrowed).
- **Error States & Messages**: No notification: "Check dashboard for updates." (Manual poll if websocket fails).
- **Loading States**: Spinner on dashboard: "Loading notifications...".

### Step 2: Accept
- **User Sees (UI Component)**: Button "Accept Mission" on mission details page.
- **Backend (API + Contract)**: Contract call: acceptMission(mission_id).
- **Error States & Messages**: Already accepted: "Mission already accepted." Tx fail: "Acceptance failed. Retry."
- **Loading States**: Spinner: "Accepting mission...".

### Step 3: IN_PROGRESS
- **User Sees (UI Component)**: Status badge: "In Progress". Optional progress updates form.
- **Backend (API + Contract)**: API updates DB status. Optional contract event for progress.
- **Error States & Messages**: N/A (state transition automatic post-accept).
- **Loading States**: Brief spinner: "Updating status...".

### Step 4: Deliver
- **User Sees (UI Component)**: Upload form for deliverables (files/IPFS). "Submit Delivery" button.
- **Backend (API + Contract)**: Upload to IPFS, then contract call: deliverMission(IPFS_hash, mission_id).
- **Error States & Messages**: Upload fail: "Delivery upload failed. Try again." 
- **Loading States**: Progress bar: "Uploading and submitting...".

### Step 5: Wait Approve/Dispute
- **User Sees (UI Component)**: Status: "Awaiting Client Approval". Timer if applicable.
- **Backend (API + Contract)**: Listens for Approve or Dispute event.
- **Error States & Messages**: Timeout: "Approval window expired. Funds auto-released."
- **Loading States**: Spinner: "Waiting for client response...".

## Flow 4: DISPUTE FLOW
Client opens dispute, 48h window, provider responds, admin arbitration, resolution (slash or release).

### Step 1: Client Ouvre Dispute
- **User Sees (UI Component)**: Button "Open Dispute" on mission page post-delivery. Form: reason, evidence.
- **Backend (API + Contract)**: Contract call: openDispute(mission_id, reason_IPFS).
- **Error States & Messages**: Window closed: "Dispute window has passed." Tx fail: "Dispute opening failed."
- **Loading States**: Spinner: "Opening dispute...".

### Step 2: 48h Window
- **User Sees (UI Component)**: Timer: "Provider has 48h to respond."
- **Backend (API + Contract)**: Contract tracks timestamp; API polls or listens for events.
- **Error States & Messages**: N/A (passive state).
- **Loading States**: N/A.

### Step 3: Provider Répond
- **User Sees (UI Component)**: Form on dashboard: response, evidence. "Submit Response" button.
- **Backend (API + Contract)**: Upload to IPFS, contract call: respondDispute(mission_id, response_IPFS).
- **Error States & Messages**: Time expired: "Response window closed." 
- **Loading States**: Spinner: "Submitting response...".

### Step 4: Admin Arbitration
- **User Sees (UI Component)**: Status: "Under Arbitration". Admin panel (internal): review evidence, decide.
- **Backend (API + Contract)**: API notifies admin. Admin triggers contract: arbitrate(mission_id, decision).
- **Error States & Messages**: Admin error: "Arbitration delayed. Contact support."
- **Loading States**: Spinner: "Awaiting admin decision...".

### Step 5: Resolution Slash ou Release
- **User Sees (UI Component)**: Notification: "Dispute Resolved: Funds [released/slashed]".
- **Backend (API + Contract)**: Contract emits Resolution event; escrow releases/slashing.
- **Error States & Messages**: Resolution fail: "Funds transfer error. Check blockchain."
- **Loading States**: Spinner: "Processing resolution...".

## Flow 5: FIAT OFFBOARDING (Provider)
Provider requests withdraw, verifies 7-day holdback, KYC if >$1K, transfers USDC, confirmation.

### Step 1: Demande Withdraw
- **User Sees (UI Component)**: Withdraw form on dashboard: amount, fiat details. "Request Withdraw" button.
- **Backend (API + Contract)**: API validates amount <= available. Records request in DB.
- **Error States & Messages**: Insufficient funds: "Available balance: X. Request less." 
- **Loading States**: Spinner: "Submitting request...".

### Step 2: Vérif Holdback 7j
- **User Sees (UI Component)**: Status: "Holdback Period: 7 days remaining."
- **Backend (API + Contract)**: API starts timer; checks contract for any disputes.
- **Error States & Messages**: Active disputes: "Cannot withdraw during active disputes."
- **Loading States**: Timer loading: "Verifying holdback...".

### Step 3: KYC Check si > $1K
- **User Sees (UI Component)**: If >$1K: KYC form (ID upload, address). "Submit KYC" button.
- **Backend (API + Contract)**: API integrates with KYC provider (e.g., Persona). Verifies and stores result.
- **Error States & Messages**: KYC fail: "KYC verification failed. Please resubmit documents." Not required if <$1K.
- **Loading States**: Spinner: "Verifying KYC...".

### Step 4: Transfer USDC
- **User Sees (UI Component)**: Confirmation: "Transferring USDC to your wallet."
- **Backend (API + Contract)**: Contract call: withdrawUSDC(amount, provider_address). Or off-chain transfer if fiat.
- **Error States & Messages**: Transfer fail: "USDC transfer failed. Check wallet."
- **Loading States**: Spinner: "Transferring funds...".

### Step 5: Confirmation
- **User Sees (UI Component)**: Success page: "Withdraw Complete!" with tx details.
- **Backend (API + Contract)**: Listens for Withdraw event; updates balance.
- **Error States & Messages**: Event timeout: "Confirmation delayed. Verify on explorer."
- **Loading States**: Spinner: "Confirming withdraw...".
