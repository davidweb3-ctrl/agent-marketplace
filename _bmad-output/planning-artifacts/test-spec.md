# Test Specification for AI Agent Marketplace on Base L2

This document provides a comprehensive test specification for the AI Agent Marketplace built on Base L2. The stack includes:

- **Smart Contracts**: Solidity 0.8.28 with Hardhat for testing (using Chai Matchers).
- **API**: Fastify 5 with Vitest and Supertest for unit/integration tests.
- **E2E**: Playwright for end-to-end browser tests.

The specification covers unit, integration, stress, edge case, and E2E tests. It includes exact test names, descriptions, and boundary values where applicable. Test files are named and structured for easy copying into projects (e.g., under `test/` for contracts, `tests/api/` for API, `tests/e2e/` for E2E, and `fixtures/` for data).

## 1. Smart Contract Tests (Hardhat with Chai Matchers)

Tests are written in Hardhat using Chai Matchers. Each contract has its own test file under `test/contracts/`. Deployment scripts and helpers are in `test/helpers/deploy.js`. Use `ethers` for interactions and `chai` for assertions.

### File: test/contracts/AgentRegistry.test.ts

- **Happy Path**
  - `should register a new agent successfully`: Verifies that an agent can be registered with valid metadata (e.g., name, capabilities) and emits `AgentRegistered` event.
  - `should update agent details by owner`: Checks that the agent owner can update metadata without errors.

- **Edge Cases**
  - `should revert on overflow in agent ID counter (max uint256)`: Tests registration when agent count reaches 2^256 - 1; expects revert with "Overflow error".
  - `should revert on underflow in agent score subtraction`: Attempts to subtract score below 0; expects revert with "Underflow error".
  - `should prevent reentrancy in registerAgent function`: Simulates reentrancy attack via a malicious contract calling back; expects revert with "ReentrancyGuard".
  - `should enforce access control for updateAgent (only owner)`: Non-owner tries to update; expects revert with "Ownable: caller is not the owner".

- **MEV/Front-Running Tests**
  - `should prevent front-running in agent registration using commit-reveal`: Tests commit phase (hash submission) and reveal phase; verifies that front-running a reveal transaction fails due to timestamp checks (e.g., 1-block delay).
  - `should handle MEV in score updates with commit-reveal`: Simulates sandwich attack on score update; ensures commit-reveal prevents manipulation.

- **Stress Tests**
  - `should handle registration of 1000 agents`: Registers 1000 agents in a loop; verifies gas usage < 10M and all registrations succeed.
  - `should manage 500 concurrent score updates`: Updates scores for 500 agents; checks for no overflows and correct event emissions.

### File: test/contracts/MissionEscrow.test.ts

- **Happy Path**
  - `should create a new mission escrow successfully`: Creates escrow with valid amount (e.g., 1 ETH) and emits `EscrowCreated` event.
  - `should release funds to agent on completion`: Completes mission and releases funds; verifies balance transfer.

- **Edge Cases**
  - `should revert on overflow in escrow amount (max uint256)`: Deposits 2^256 ETH; expects revert with "Overflow in deposit".
  - `should revert on underflow in partial release`: Attempts to release more than deposited (e.g., release 2 ETH from 1 ETH escrow); expects "Insufficient funds".
  - `should prevent reentrancy in releaseFunds function`: Malicious agent contract reenters; expects revert with "Reentrancy detected".
  - `should enforce access control for dispute resolution (only oracle)`: Non-oracle calls resolveDispute; expects "Access denied".

- **MEV/Front-Running Tests**
  - `should prevent front-running in mission completion using commit-reveal`: Commits completion hash, then reveals; tests that front-running reveal fails due to nonce check.
  - `should mitigate MEV in fund deposits`: Simulates flash loan attack on deposit; ensures commit-reveal locks the amount.

- **Stress Tests**
  - `should handle creation of 500 mission escrows`: Creates 500 escrows with varying amounts; verifies all states are correct.
  - `should process 200 concurrent releases`: Releases funds from 200 escrows; checks gas limits and no failures.

### File: test/contracts/AGNTToken.test.ts

- **Happy Path**
  - `should mint tokens to a new address`: Mints 1000 AGNT to user; verifies balance and `Transfer` event.
  - `should transfer tokens between users`: Transfers 500 AGNT; checks balances.

- **Edge Cases**
  - `should revert on overflow in total supply (max uint256)`: Mints up to 2^256 - 1, then one more; expects "Supply overflow".
  - `should revert on underflow in transfer (insufficient balance)`: Transfers 1001 from 1000 balance; expects "Insufficient balance".
  - `should prevent reentrancy in transferFrom`: Malicious approve/transfer reenters; expects revert.
  - `should enforce access control for minting (only minter role)`: Non-minter calls mint; expects "MinterRole: caller does not have role".

- **MEV/Front-Running Tests**
  - `should prevent front-running in token approvals using commit-reveal`: Commits approval hash, reveals; tests front-running failure.
  - `should handle MEV in batch transfers`: Simulates sandwich on multi-transfer; ensures atomicity.

- **Stress Tests**
  - `should mint and transfer for 1000 users`: Mints to 1000 addresses, then transfers; verifies no gas exceedance.
  - `should handle 500 concurrent approvals`: Approves for 500 users; checks allowances.

### File: test/contracts/ReputationOracle.test.ts

- **Happy Path**
  - `should update reputation score successfully`: Updates score to 90; emits `ScoreUpdated`.
  - `should query reputation accurately`: Queries score for agent; returns correct value.

- **Edge Cases**
  - `should revert on overflow in score aggregation (max uint256)`: Aggregates scores exceeding 2^256; expects overflow revert.
  - `should revert on underflow in score decay`: Decays score below 0; expects "Score cannot be negative".
  - `should prevent reentrancy in updateScore`: Reentrant call; expects revert.
  - `should enforce access control for oracle updates (only authorized)`: Unauthorized update; expects "OracleRole: unauthorized".

- **MEV/Front-Running Tests**
  - `should prevent front-running in score submissions using commit-reveal`: Commits score, reveals; tests MEV resistance.
  - `should mitigate MEV in batch updates`: Sandwich attack on multi-score update; ensures integrity.

- **Stress Tests**
  - `should update scores for 1000 agents`: Updates 1000 scores; verifies performance.
  - `should query 500 reputations concurrently`: Queries 500 times; checks response times.
